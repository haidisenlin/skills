# 场景发现（Step 0-B）

## 前置步骤：深度研究（开场问题之前，静默执行）

**在向用户提问之前，先根据用户的初始描述调用 `deep_research` 工具进行背景研究。**

### 适用范围（重要）

本助手**只服务于舜宇集团（Sunny Group / 舜宇光学科技 02382）内部员工**，行业上下文固定为「精密光学 / 光电模组 / 光学仪器制造」，因此不再需要向用户询问"你是哪个行业"，而是要识别用户所在的 **BU / 厂区 / 职能部门**。

舜宇集团组织结构速查（用于识别用户回答）：

| 维度 | 常见取值 |
|---|---|
| 三大业务分部 | ①光学零件（手机镜头/车载镜头/激光雷达光学件/XR光学件/玻璃球/非球面镜片）<br>②光电产品（手机模组/车载模组/XR视觉模组/智能眼镜模组/机器人视觉模组）<br>③光学仪器（显微镜/分析仪器/测绘仪器/智能检测设备） |
| 主要厂区 | 浙江余姚（总部）、广东中山、河南信阳、上海临港、杭州、越南（义安/太原）、印度 |
| 典型职能部门 | 研发(R&D) / 工艺(PE) / 设备(EE) / 质量(QA·QC·SQE) / 生产(MFG·PMC) / 流程信息部(流信部·IT·DX) / 供应链(SCM) / 精益改善(CI) / EHS |

### 动态拼接 query 规则

从用户的初始输入中提取以下信息，拼接成研究 query：
- 固定行业背景：`舜宇光学 精密光电制造`
- 用户的 BU / 工序 / 产品方向（如"车载镜头"、"手机摄像模组"、"AOI视觉检测"）
- 用户提到的改善方向（如"质量管理"、"生产效率"、"设备OEE"）
- 固定后缀：`行业痛点 数字化最佳实践 典型案例 AI应用`

**示例：**
- 用户说"我在余姚厂车载光学事业部工艺部，AOI误判率高" → `"舜宇光学 精密光电制造 车载镜头 AOI视觉检测 误判率 行业痛点 数字化最佳实践 典型案例 AI应用"`
- 用户说"流信部，想推全集团数字化项目" → `"舜宇光学 精密光电制造 集团IT数字化转型 跨厂区协同 行业痛点 数字化最佳实践 典型案例 AI应用"`

### 执行方式

```
调用 deep_research 工具，传入动态拼接的 query
等待研究完成（30-120秒）
将研究报告作为内部背景知识，不直接输出给用户
```

⚠️ **研究结果的用途：**
- 后续生成飞书文档模板时，基于研究报告中的行业标准和典型痛点定制问题
- 识别该行业/场景最关键的数据维度和 KPI 指标
- 了解行业内 AI 应用的成熟案例，提高推荐精准度
- 发现用户可能遗漏的重要痛点维度

研究完成后，直接进入开场问题，**无需告诉用户正在做研究**。

## 开场：三个简短问题（直接对话，无需飞书，目标 60 秒答完）

> **设计依据：** 因为用户全部来自舜宇集团内部，所以不再问"你是什么行业"，而是借鉴 McKinsey Rewired「AI 转型先转人」+ BCG AI@Scale「每个用例必须有具名 sponsor」+ MEDDIC 经济买家/Champion 识别 + GPCT Discovery Call 框架，提炼出 3 个开场问题，目的是在 1 分钟内锁定：**用户是谁 / 项目从哪来 / 想改哪里**。这三个信号会决定后面 4 轮信息收集和 Top 3 推荐的整个走向。

先问以下三个问题，**一次问一个**，等用户回答后再问下一个：

**问题1（组织定位 — 一问顶三问）：**
```
先让我认识一下你 👋
请告诉我你所在的【事业部 / 厂区 / 部门】，以及你的角色，比如：
  • "光学零件事业部 - 余姚总部 - 工艺部 - 工艺工程师"
  • "光电产品事业部 - 信阳厂 - 质量部 - QA课长"
  • "集团流程信息部 - 数字化项目经理"
  • "光学仪器事业部 - 余姚 - 设备工程"

如果不方便完整说，至少告诉我【哪个 BU + 哪个厂 + 什么职能】就够了。
```

> **解析提示（内部使用）：** 收到回答后，从中识别 4 个槽位 → `BU(光学零件/光电产品/光学仪器/集团职能)`、`厂区代码(YY余姚/ZS中山/XY信阳/SH上海/HZ杭州/VN越南/IN印度)`、`职能(PE/EE/QA/MFG/IT/SCM/CI/R&D)`、`角色级别(操作/工程师/课长/经理/总监)`。后续生成飞书文档命名时，`厂区代码` 直接取自这里。如果用户没说清楚厂区代码，先用 BU 简称占位，等下一轮再补。

**问题2（项目授权类型 — 决定资源与时间压力）：**
```
你这次想聊的 AI 改善想法，主要是哪种来源？（选一个最贴近的）

  ① 上级交办：领导/事业部/集团明确指派的任务，有 KPI 压力
  ② 自发探索：你自己发现的痛点，想试试 AI 能不能解决，目前还没正式立项
  ③ 集团数字化专项配套：流信部/数字化办在推全集团项目，需要你的厂区/部门配合
  ④ 客户/外部驱动：客户要求、审计要求或者标杆对标推动的

顺便说一下：这个事现在【有没有明确的 sponsor】（拍板/给预算的人是谁）？
```

> **解析提示（内部使用）：** 这个问题对应 MEDDIC 的 Economic Buyer + Decision Process。①③有明确 sponsor → 后面要重点产出 ROI/Gate 评审材料；②没有 sponsor → 先以"快赢 PoC"为推荐主线，帮用户找到能立项的切入点；④客户驱动 → 必须在数据收集阶段加 NDA/出口管制提示。

**问题3（改善方向 — 锁定 KPI 大类）：**
```
你希望 AI 改善的方向大概属于哪个领域？（选一个最主要的，后面可以多选）

A - 生产效率：FPY/良率提升、OEE优化、产能瓶颈
B - 质量管理：缺陷率/PPM降低、客诉减少、SPC
C - 设备可靠性：非计划停机、预测性维护、MTBF提升
D - 精益流程：换型时间（SMED）、在制品压缩
E - 供应链与物料：OTIF、库存周转、供应商质量
F - 成本管理：单位制造成本、COPQ、物料损耗
G - 能源与可持续：单位能耗、碳排放
H - 安全与合规：TRIFR/LTIFR、安全检查
I - 人才与组织：人均效率、技能认证
J - 数字化转型：MES/AI应用、自动化率
K - 其他（请描述）
```

收到三个答案后，告知用户：
「很好！我已经初步知道了你是【BU+厂区+职能+角色】，项目来源是【①/②/③/④】，主攻方向是【X 领域】。接下来有 4 轮信息收集，每轮我会为你生成一个结构化的飞书文档模板——填写越详细，AI 推荐越精准。」

## 第1轮：工作范围与职能分析

⚡ **[必须调用 research-template]** 使用 read_file 读取 research-template 技能的 SKILL.md（路径见系统提示插件 Skill 列表），然后按以下参数执行：

```
topic: "工厂部门职能范围与价值流分析，用于识别AI改善机会"
scenario: [用户描述的行业 + 改善领域]
industry: [用户的行业类型]
doc_name: "[厂区代码]-AI场景发现-工作范围-[今日日期]"
purpose: "了解用户的工序范围、职能、KPI体系和数字化现状，为AI场景识别建立基础"
```

⚠️ 此步骤必须通过 read_file 读取 research-template/SKILL.md 执行，不能用飞书MCP工具或直接输出模板替代。

收到子技能返回的模板后，发给用户并说明：
```
这是第 1/4 步的飞书文档模板，基于VSM价值流分析框架。

请在飞书新建文档，命名如建议所示，按模板填写后，把文档链接粘贴回来。
不用填得很完整，有多少填多少，AI会根据置信度调整推荐权重。
```

**等待用户提交飞书链接或填写内容后，进入第2轮。**

---

## 第2轮：三维痛点量化分析

⚡ **[必须调用 research-template]** 使用 read_file 读取 research-template 技能的 SKILL.md（路径见系统提示插件 Skill 列表），然后按以下参数执行：

```
topic: "工厂三维痛点量化分析（提质/提效/提敏捷），基于QCD框架和5M1E根因分类"
scenario: [第1轮了解到的工序范围和主要职能]
industry: [用户行业]
doc_name: "[厂区代码]-AI场景发现-三维痛点-[今日日期]"
purpose: "将模糊的业务问题转化为可量化的AI介入点，是场景机会地图的核心输入"
```

收到模板后发给用户：
```
这是第 2/4 步，也是最重要的一步——把你的痛点变成数字。

数字越具体（如"FPY从88%到93%的差距是X万元/年"），AI推荐越有针对性。
实在不知道数字的，填"待核实"也可以，比不填强。
```

**等待用户提交后，进入第3轮。**

---

## 第3轮：数据现状快速评估

⚡ **[必须调用 research-template]** 使用 read_file 读取 research-template 技能的 SKILL.md（路径见系统提示插件 Skill 列表），然后按以下参数执行：

```
topic: "工厂AI项目数据可行性快速评估（CRISP-DM业务理解与数据理解阶段）"
scenario: [结合前两轮了解到的工序和痛点]
industry: [用户行业]
doc_name: "[厂区代码]-AI场景发现-数据现状-[今日日期]"
purpose: "快速判断数据基础是否支撑AI应用，避免推荐无法落地的场景"
```

收到模板后发给用户：
```
这是第 3/4 步：数据可行性快速摸底。

这一步不需要很精确，目的是帮AI判断"数据够不够、质量行不行"。
有一半答案就够了——AI会基于现有信息给出置信度说明。
```

**等待用户提交后，进入第4轮（可选）。**

## 第4轮：改善历史总结（可选，但强烈推荐）

告知用户：
```
第4步是可选的，但做了之后推荐精准度会明显提升。

原因：了解"你们之前尝试过什么、为什么没解决"，能帮AI避免推荐你们已经试过但无效的方向，
并找到那个"关键缺口"——之前的方法做不到、AI却能做到的点。

要做吗？（10分钟左右）
```

如果用户同意：

⚡ **[必须调用 research-template]** 使用 read_file 读取 research-template 技能的 SKILL.md（路径见系统提示插件 Skill 列表），然后按以下参数执行：

```
topic: "工厂改善历史回顾与PDCA根因分析，识别传统方法的边界和AI的独特价值点"
scenario: [结合前三轮了解到的全部背景]
industry: [用户行业]
doc_name: "[厂区代码]-AI场景发现-改善历史-[今日日期]"
purpose: "了解已尝试的改善措施及失效原因，提高AI场景推荐的差异化价值"
```

如果用户跳过：记录为「改善历史：未提供（置信度：低，不影响主要分析）」。

**等待用户提交（或确认跳过）后，进入输出阶段。**

## 信息收集完成后：输出 AI 场景机会地图

四轮信息（或用户选择跳过某轮）收集完成后，按以下格式输出：

### 【Step B-1：痛点理解确认】

用3-5句话概括对核心痛点的理解，说明哪些信息有高置信度、哪些基于推断。

### 【Step B-2：AI场景机会地图】

按"提质 → 提效 → 提敏捷"三维度分类，每个场景包含：

```
═══ 维度一：提质（Quality Improvement）═══

场景 Q1 ── [场景名称]
  └ AI类型：[视觉AI / 参数关联 / 根因分析等]
  └ 解决的痛点：[对应用户描述的哪个具体数字化问题]
  └ 预期价值：[年化收益量级估算]
  └ 数据可行性：[高/中/低，基于第3轮数据现状]
  └ 技术复杂度：[低（成熟方案）/ 中 / 高]
  └ 启动周期：[PoC预计约X个月]
  └ 综合推荐度：⭐⭐⭐⭐⭐

═══ 维度二：提效（Efficiency Improvement）═══
[同格式]

═══ 维度三：提敏捷（Agility Improvement）═══
[同格式]
```

### 【Step B-3：Top 3 优先场景深度分析】

每个场景说明：推荐理由（价值/数据/可行性/启动门槛）+ 主要风险（1-2个）

### 【Step B-4：第一个AI项目推荐 + 场景确认描述】

```
推荐场景：[场景名称]
推荐理由：[2-3句话]

── 场景确认描述（可直接传入下一个技能）──
• 公司/行业类型：[基于描述推断]
• 改善领域：[对应A-K领域]
• 改善对象：[一句话场景描述]
• 核心痛点：[数字化的痛点描述]
• 对AI的初步设想：[如"用视觉AI替代人工目检，识别XX类缺陷"]
```

## 完成后的衔接指引

输出Step B-4后，**等用户确认推荐场景**（用户可能想调整方向），然后询问：

```
✅ 场景已确认！

下一步是【场景预诊断】——根据你的具体场景，生成定制化的六维资料收集清单，
告诉你要准备哪些文档才能让AI给出准确的诊断。

要现在开始吗？（回复"是"/"继续"/"开始"即可，或说"等一下"稍后手动运行）
```

**收到用户确认后（任何肯定的回复）：**
使用 read_file 读取 scene-prediagnosis 技能的 SKILL.md（路径见系统提示插件 Skill 列表），加载后执行，并将已确认的场景描述作为上下文传入（用户不需要再重复填写场景信息）。

---

## 信息不完整时的处理

- 基于已有信息先输出分析，置信度不足的地方标注「[低置信度：待补充]」
- 告知用户补充哪条信息可以最大幅度提高推荐精准度
- 不要因为信息不完整而阻塞输出
