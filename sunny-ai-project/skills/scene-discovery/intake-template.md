## 📋 Workshop 使用说明（请先看这一节）

本模板配合 **舜宇 AI 场景发现 Workshop** 使用，分两个阶段填写：

### 🏠 阶段 A：会前准备（在工位 / 家里完成，约 15 分钟）

会前请你把以下内容**先填好或想好**，因为它们要么需要查系统、要么需要翻看你的 OKR / 绩效文档、要么需要点反思时间，到了会场临时填会很仓促：

- ✅ **第一部分 · 个人定位**（5 字段，2 分钟）— 都是事实性字段
- ✅ **第三部分 3.1 Goal**（你被考核的最高目标）— 翻你的 OKR / 年度绩效文档抄过来
- ✅ **第三部分 3.3 Metrics 表格**（KPI 当前值 / 目标值 / 数据源）— **必须查系统**，会场临时编效果会很差
- ✅ **第五部分 5.1 利益相关方草稿**（Sponsor / Champion / 阻力方的姓名）— 你需要时间想 + 可能要查名单

> 💡 **填不全没关系**：如果某个字段你确实拿不到数据，写"待核实"或留空。重要的是**能填的尽量填好**，让会场时间用在更有价值的讨论上。

### 🎯 阶段 B：Workshop 会场现场填写（在会场 + facilitator 引导，约 25-40 分钟）

到会场后，facilitator 会带大家**逐节填写以下内容**，这些字段**特地留到现场**是因为：

- 🎯 **第二部分 · SIPOC 工作流程**（10-15 分钟）— SIPOC 是六西格玛的经典工具，第一次接触的人独自填会写得很粗。**facilitator 会带你一起拆解 Process 步骤**，并通过结对评审让你的 SIPOC 真正能用
- 🎯 **第三部分 3.2 Questions + 3.4 最头疼指标 + 卡住原因**（5-10 分钟）— 这是反思性内容，**结对讨论比独自闷想效果好 5 倍**
- 🎯 **第四部分 · 数字化成熟度自评**（10-15 分钟）— 独自打分时人会下意识把自己抬高 1-2 分。**会场会做小组互校准**：3-4 人一组分享打分依据，然后再修正最终分数。这一步出来的数据才有可比性
- 🎯 **第五部分 5.1 利益相关方校准 + 5.2 决策与预算**（5 分钟）— 用你会前准备的草稿做基础，facilitator 会用提问帮你发现遗漏的"隐形利益相关方"
- 🎯 **第六部分（可选）· AI Use Case Canvas**（10 分钟，时间允许时）— 如果你会前已经有 AI 想法，用这一节捕捉它；如果没有，**workshop 后半场的 AI 助手对话会帮你形成假设**，到时再回来填这一节

### 📌 给 Workshop Facilitator 的话

如果你是 facilitator（现场主持人），下方每一节顶部的标签会告诉你这一节是 🏠（会前完成）、🎯（会场带做）还是 🏠+🎯（会前打草稿+会场细化）。每个 🎯 节都附了【Workshop 现场操作】小框，里面写了具体的引导动作和时间预算。

## 第二部分 · 你的工作流程（SIPOC 框架）　🎯 **会场现场完成**

> **框架来源**：**SIPOC**（Supplier-Input-Process-Output-Customer），出自 IBM 1980s 提出、Six Sigma DMAIC「Define 阶段」的标准工具。它的好处是**不区分物料流和信息流**——可以描述一条镜片组装线，也可以描述一个合同审批流程或一份月度报表的生产过程。
>
> 这一节是**整份文档里最能让 AI 看懂你工作的部分**，请认真填，每一格都给了现场和办公两类示例。
>
> **🎯 为什么留到会场？** 大部分人没接触过 SIPOC，独自填会写得太粗（典型错误：Process 只写"做产品"或"处理需求"）。会场 facilitator 会带你逐项细化，并通过结对评审让你的 SIPOC 真正能用。

> **🎯 Workshop 现场操作（给 facilitator）** ⏱ **15 分钟**
>
> 1. **个人初稿（5 分钟）**：每人独立填一遍 SIPOC 五要素，先不追求精确
> 2. **结对评审（5 分钟）**：2 人一组互相评审，重点检查：
>    - Process 步骤是否拆到 3-7 个具体动作？（"做产品"→ "清洗→对准→点胶→AOI→包装"）
>    - Supplier 是不是只写了直接上游？有没有遗漏数据/系统类供应方？
>    - Customer 是不是只写了下游工序？最终用户是谁？
> 3. **集体校准（5 分钟）**：facilitator 挑 1-2 个典型案例（一个现场、一个办公）在白板上演示完整 SIPOC，让大家对照修正自己的版本

### 2.1 SIPOC 五要素

| 维度 | 含义 | 你的填写 | 现场示例 | 办公示例 |
|---|---|---|---|---|
| **S — Supplier 上游** | 谁/哪个部门/哪个系统给你提供工作输入？ | | 玻璃球毛坯供应商 ABC<br>+ 上一道工序（清洗工序） | 业务部门（提报数字化需求）<br>+ 集团 HR（提供人事数据） |
| **I — Input 输入** | 你接收什么？数量级/频率多少？ | | 毛坯镜片，月供 50 万颗<br>+ 工艺参数表 | 需求文档（每周 ~5 份）<br>+ 月度运营数据（10 张表）|
| **P — Process 流程** | 你执行哪些活动？请用 3-7 个动作步骤描述 | | 清洗 → AA 对准 → 点胶 → 固化 → AOI 复检 → 包装 | 需求评审 → 技术方案设计 → 开发 → UAT → 上线 → 运维监控 |
| **O — Output 输出** | 你交付什么？数量级/频率多少？ | | 合格的镜头模组，月产 X 万颗 | 上线的系统功能 / 月度报表 / 招聘人数 / 风险评估意见 |
| **C — Customer 下游** | 谁/哪个部门接收你的输出？他们怎么用？ | | 余姚 AA 工序，节拍 X 秒/件 | 业务用户 / 工厂端 / 集团总部 / 外部客户 |

### 2.2 你这个流程在更大价值链中的位置（勾选一个）

> **框架来源**：Porter Value Chain（迈克尔·波特价值链模型，1985）。

- [ ] **核心运营**（直接产生外部客户价值：制造/研发/销售）
- [ ] **物流/供应链**（入向或出向物流、库存、配送）
- [ ] **支持职能-技术**（IT、数据、自动化、流程数字化）
- [ ] **支持职能-人才**（HR、培训、组织发展）
- [ ] **支持职能-治理**（财务、法务、合规、内审）
- [ ] **支持职能-基础设施**（厂房、设备、EHS、IT 基础设施）

## 第三部分 · 你的 KPI / OKR 体系（GQM 框架）　🏠+🎯 **混合（重点字段会前必做）**

> **框架来源**：**GQM = Goal-Question-Metric**，由 Basili & Rombach 在 1994 年提出，原本用于软件工程度量，现已成为管理科学公认的指标设计方法。它强迫你**先想清楚目标和你要回答的问题，再选指标**——避免"为了 KPI 而 KPI"的陷阱。
>
> **填写时机分配：**
> - 🏠 **3.1 Goal** + **3.3 Metrics 表格**：**会前必做**——这两个需要你翻 OKR 文档、查系统抓数据，会场临时编出来的数字毫无价值
> - 🎯 **3.2 Questions** + **3.4 最头疼指标**：会场结对讨论完成

### 3.1 Goal — 你被考核的最高目标是什么？　🏠 **会前完成**

> 用 1-2 句话写出来。如果你有多个并列目标，挑最重要的那个。
>
> **🏠 会前操作**：直接打开你的年度 OKR / 绩效目标文档抄过来。如果没有正式 OKR，写你年初向上级承诺的最重要目标。

```
[在此填写。示例：]

[现场示例] 在 2026 年内把车载镜头组装线 FPY 从 87% 提升到 92%，
            同时单位制造成本降低 8%。

[办公示例] 在 2026 年内把集团 HR 招聘流程的"岗位填充周期"从 45 天
            缩短到 25 天，同时离职率降至 12% 以下。
```

### 3.2 Question — 围绕这个目标，你想回答哪些关键问题？　🎯 **会场完成**

> 列 2-3 个。这些问题就是你工作中最常被卡住的地方。
>
> **🎯 Workshop 现场操作（给 facilitator）** ⏱ 5 分钟
> 1. 提示：好的 Question 应该是"如果我能回答这个，我就知道该怎么改进"——**带"为什么/哪些/什么时候"**这样的疑问词
> 2. 反例：不要写"提升良率"（这是 Goal，不是 Question）
> 3. 正例：" 哪几个工序贡献了 80% 的不良？"、"哪些岗位的简历筛选最不准？"
> 4. **结对分享**：和你旁边的同事互相念 2-3 个 Question，对方来评判是否符合"可回答"的标准

```
[示例：]
1. 现在影响 FPY 的主要缺陷类型是什么？哪些是可以预防的？
2. 哪几台设备在哪些参数组合下最容易产生不良？
3. 同样的物料和参数，为什么不同班次的良率差距这么大？

[办公示例：]
1. 哪些岗位最难招？卡在简历筛选还是面试还是 offer 接受？
2. 高潜员工离职前 6 个月的行为信号是什么？
3. 不同 BU 的招聘周期差距，是流程问题还是人才市场问题？
```

### 3.3 Metric — 用什么指标来回答这些问题？　🏠 **会前完成（必须查系统）**

> **🏠 会前操作**：这是整份模板里**最不能临时编**的部分。请打开你日常用的 MES / ERP / SAP / HR 系统 / 飞书表格，把当前值和目标值的真实数字抄下来。会场临时填的"差不多 87%"和系统里实际的 86.4% 会让 AI 推荐的 ROI 估算偏差 30% 以上。
>
> 如果某个指标你连数据源都不知道，写"数据源待查"——这本身就是一个有价值的诊断信号。

| 指标名称 | 单位 | 当前值 | 目标值 | 数据来源 | 复盘频率 |
|---|---|---|---|---|---|
| 例：FPY | % | 87.3 | 92 | MES 自动 | 周 |
| 例：岗位填充周期 | 天 | 45 | 25 | HR 系统 | 月 |
|  |  |  |  |  |  |
|  |  |  |  |  |  |
|  |  |  |  |  |  |

### 3.4 这些指标里你最头疼的是哪一个？为什么解决不了？　🎯 **会场完成**

> **这是 AI 最关心的字段**——卡住的根本原因是什么？已经试过哪些方法？
>
> **🎯 Workshop 现场操作（给 facilitator）** ⏱ 8 分钟
> 1. **个人写 3 分钟**：基于你 3.3 表格里的指标，挑一个最头疼的，写："为什么解决不了？我已经试过 X、Y、Z 但都没用。"
> 2. **3 人小组分享 5 分钟**：每人 1.5 分钟分享自己的最头疼指标 + 已尝试方案。组员可以追问"为什么 Y 没用？"——这种追问往往暴露出"其实我没真的试过 Y"或"Y 失败的原因是 W，那 AI 能不能解决 W？"
> 3. **关键产出**：每个人都要带着一个**"传统方法做不到的具体卡点"** 离开这一节，这是后面 AI 推荐的核心切入点

```
[在此填写。示例：]

[现场] FPY 卡在 87%-88% 半年没动。已经做过 SPC 收紧（无效）、
       来料筛选（成本高）、工艺参数 DOE（提升 0.3%）。
       我怀疑问题在不同班次操作员的手法差异，但缺乏数据证明。

[办公] 高潜员工离职率从去年 8% 涨到 14%。HR 做了离职面谈，
       但拿到的多是泛泛的"个人原因"，没有可操作的根因分析。
       想用 AI 看看能不能从员工日常协作行为/绩效曲线里
       提前 3-6 个月识别离职风险信号。
```

## 第四部分 · 数字化成熟度自评（Gartner / INCIT 框架简化版）　🎯 **会场完成（关键互校准）**

> **框架来源**：**Gartner Digital Business Maturity Model** + **INCIT AIMRI（AI Maturity & Readiness Index）工业 AI 成熟度模型**。完整版有 30+ 维度，这里压缩为 5 个最关键的，每个维度做 1-5 级自评。
>
> **🎯 为什么留到会场？** 数字化成熟度自评有一个公认的问题：**人独自打分时会不自觉地把自己抬高 1-2 分**（学术上叫 self-serving bias）。Gartner 自己的研究显示，没有校准的自评数据噪声大到几乎不可用。会场会做"小组互校准"，让数据真正可比。

> **🎯 Workshop 现场操作（给 facilitator）** ⏱ 12 分钟
>
> 1. **个人初评（3 分钟）**：每人独自给 5 个维度各打一个分，**先不要看周围人的分**
> 2. **3-4 人小组互校准（6 分钟）**：
>    - 每个人轮流报"我在维度 X 打了 N 分，因为……"（30 秒）
>    - 组员追问："你说数据采集打了 4 分，那你能现场举一个'全自动从源头采集'的具体例子吗？"
>    - 如果举不出实例，主动把分数下调
>    - **目标**：让每个分数都有具体证据支撑，而不是模糊感觉
> 3. **个人修正（3 分钟）**：基于小组讨论修正你的最终分数
> 4. **可选**：小组在白板上贴出每人 5 维分数的雷达图，大家直观看到自己组的差异

## 第五部分 · 利益相关方与决策权（RACI + MEDDIC 框架）　🏠+🎯 **混合（会前打草稿，会场校准）**

> **框架来源**：
> - **RACI 矩阵**（Responsible-Accountable-Consulted-Informed），项目管理经典工具
> - **MEDDIC**（Metrics-Economic Buyer-Decision Criteria-Decision Process-Identify Pain-Champion），原本是 PTC 公司的 B2B 销售方法论，用在内部立项里能精准识别"谁说了算"
>
> AI 项目立项最常见的失败原因不是技术不行，而是**没有找对那个能拍板和给资源的人**。这一节强制你想清楚这件事。
>
> **填写时机分配：**
> - 🏠 **会前**：5.1 角色矩阵的姓名要靠你自己想 + 翻通讯录，**会前先打个草稿**
> - 🎯 **会场**：facilitator 用提问帮你**发现遗漏的"隐形利益相关方"**（典型遗漏：流信部、法务、采购、最终客户）

> **🎯 Workshop 现场操作（给 facilitator）** ⏱ 5 分钟
>
> 把你会前填的角色矩阵摊开，facilitator 会带你做"隐形利益相关方扫描"，问你 4 个问题：
> 1. **数据问题**：这个项目用的数据归谁管？流信部 / DBA / 数据治理委员会有没有列在协作方？
> 2. **合规问题**：这个项目会不会涉及客户数据 / 员工数据 / 财务数据？法务 / 内审 / 合规有没有在表里？
> 3. **采购问题**：会不会需要外部供应商 / 软件采购？采购部门有没有在表里？
> 4. **下游问题**：项目交付后谁会真的用？把"主要受益方"具体到岗位名而不是"业务部门"
>
> **关键产出**：5.1 表格里至少有 5 个具名的角色，每个角色都有姓名或岗位

## 📚 本模板使用的方法论汇总

| 部分 | 框架 | 出处 |
|---|---|---|
| 第二部分 工作流程 | **SIPOC** | Six Sigma DMAIC「Define 阶段」标准工具 |
| 第二部分 价值链定位 | **Porter Value Chain** | Michael Porter《Competitive Advantage》(1985) |
| 第三部分 KPI 设计 | **GQM (Goal-Question-Metric)** | Basili & Rombach (1994), Software Engineering Lab |
| 第四部分 数字化成熟度 | **Gartner Digital Business Maturity** + **INCIT AIMRI** | Gartner 2024 + 国际制造业创新中心 |
| 第五部分 利益相关方 | **RACI 矩阵** + **MEDDIC** | PMI 标准 + PTC 销售方法论 |
| 第六部分 AI 用例 | **Bernard Marr AI Use Case Canvas** | 《Data Strategy》(Marr, 2021) |

> 这些框架不是装饰——AI 助手在后续诊断时会**显式引用**它们的字段名。例如你在 SIPOC 里填的 "Process 流程"会直接驱动 5M1E 痛点分解；GQM 里的 "最头疼指标"会成为 Top 3 推荐的优先级权重。**你填得越认真，AI 给你的推荐就越精准。**
