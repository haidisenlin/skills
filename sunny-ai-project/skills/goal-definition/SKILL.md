# 目标定义（阶段 1）

---

**方法论来源：** Hoshin Kanri（方针管理）目标展开 · SMART目标框架 · OEE/DMAIC/TPM/VSM/ISO 50001等KPI体系 · WEF Lighthouse Factory KPI数据库 · Google OKR信心指数理论

## 你的任务：分四步引导用户完成目标定义

### 第一步：改善领域选择（1.0）

先展示十大改善领域，让用户选择（可多选，建议主领域≤2个）：

```
请选择最符合你项目的改善领域（可多选）：

A - 生产效率：良率/FPY提升、OEE优化、产能瓶颈、节拍提速
B - 质量管理：缺陷率/PPM降低、客诉减少、SPC实施
C - 设备可靠性：非计划停机减少、预测性维护、MTBF提升
D - 精益流程：换型时间缩短（SMED）、在制品压缩、价值流优化
E - 供应链与物料：OTIF、库存周转、供应商质量
F - 成本管理：单位制造成本、COPQ、物料损耗
G - 能源与可持续：单位能耗、碳排放、废料率
H - 安全与合规：TRIFR/LTIFR、未遂事件、安全检查
I - 人才与组织：人均效率、技能认证率、改善提案数
J - 数字化转型：MES/AI/ML应用、自动化率提升
K - 其他自定义（描述你的场景）
```

用户选择后，加载对应的KPI框架（见支撑文件 [kpi-reference.md](kpi-reference.md)）。

### 第三步：目标合理性自检——雄心度-现实校准（1.3）⚠️ 核心防护网

**这是文档中最重要的防护环节。** 最常见的失败模式：管理层要求年底FPY从85%提到98%，但数据还没接通、MSA没做、AI工程师还没到位。

**A. 基础合理性核查（让用户逐项回答）：**

| 自检项 | 用户回答 |
|---|---|
| 基线数据是实测值，不是估算 | 是/否 |
| 目标值参考过行业基准或竞争对手数据 | 是/否/不清楚 |
| 改善幅度在6-24个月内有行业先例可循 | 是/否（属于突破性目标）|
| 即使只实现60%的目标，ROI仍然为正 | 是/否/未估算 |
| 目标已经过有决策权的管理层正式认可 | 书面认可/口头认可/尚未沟通 |

> 如果有2项以上"否"，在诊断交互时重点验证目标合理性。

**B. 目标雄心度分级（必填）：**

先引导用户计算改善幅度：
```
改善幅度 = (目标值 - 基线值) ÷ 基线值 × 100%
示例：FPY从85%→92% → 改善幅度 = (92-85)÷85×100% ≈ 8.2%
```

然后对照分级表确认雄心度：

| 改善幅度 | 行业数据支撑 | 雄心度分级 |
|---|---|---|
| < 10% | 制造AI第一年普遍可达 | 🟢 保守 |
| 10–25% | WEF Lighthouse工厂中位数 | 🟡 适中（推荐首选） |
| 25–40% | 需②③④三维度均达3-4级；典型周期18-24个月 | 🟠 激进 |
| > 40% | 仅顶尖10%灯塔工厂实现，需24-36个月 | 🔴 变革性 |

**C. OKR信心指数（必填）：**

问用户：**"如果让你打赌，在设定的时间节点内完整实现这个KPI目标的概率有多大（1-10分）？"**

| 信心分数 | 诊断信号 | 后续动作 |
|---|---|---|
| 8-10分 | 目标可能过于保守 | 考虑适当提高挑战性 |
| 5-7分 | ✅ 理想区间 | 正常推进 |
| 3-4分 | ⚠️ 目标偏激进 | 诊断交互后AI给出校准方案 |
| 1-2分 | ❌ 目标与现实严重错位 | 诊断交互后强制输出校准评估 |

**触发规则（满足任一条，诊断时触发专项校准）：**
- 雄心度分级为🟠激进或🔴变革性
- OKR信心分数≤4分
- 1.3-A有2项以上"否"

### 第四步：约束条件与项目背景（1.4 + 1.5）

**约束条件（逐项了解）：**

| 约束类型 | 询问用户 |
|---|---|
| 停机/实验窗口 | 可随时停机/需提前计划（多少天）/不可停机 |
| 改造与采购预算 | 无上限/上限¥___万/待审批/无预算 |
| 参数/流程修改权限 | 工程师可自行决定/需主管审批/需客户批准/不允许修改 |
| 客户/认证体系约束 | 有管控参数（IATF/REACH/FDA）/无/不确定 |
| IT/OT网络与数据安全 | 允许AI系统访问设备数据/需IT审批/严格隔离 |
| 人员约束 | 团队稳定/高流动率/技能明显不足/外包团队 |

**项目背景（飞书文档）：**

告知用户完成KPI矩阵后再进行此步骤。

⚡ **[必须调用 research-template]** 使用 read_file 读取 research-template 技能的 SKILL.md（路径见系统提示插件 Skill 列表），然后按以下参数执行：

```

industry: [用户行业]
doc_name: "[厂区代码]-[改善项目简称]-项目背景说明-[今日日期]"
purpose: "为后续AI诊断和方案设计提供项目背景和约束条件上下文"
```

⚠️ 此步骤必须通过 read_file 读取 research-template/SKILL.md 执行，不能用飞书MCP工具或直接输出模板替代。

收到子技能返回的模板后，发给用户并说明：

```
请在飞书新建文档，按返回的模板填写。这部分内容很重要，能帮AI理解
为什么要做这个项目、有什么限制条件、期望是什么。
```
