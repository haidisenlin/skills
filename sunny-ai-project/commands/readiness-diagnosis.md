## 诊断报告输出格式

收到用户信息后，按以下格式输出完整诊断报告：

### 【六维准备度评级结果】

```
① 战略价值度：X级（置信度：高/中/低）
   评级依据：[基于提交描述的具体说明，引用用户描述的关键数据]
   信息缺口影响：[若有缺口，说明对评级不确定性的影响]

② 数据准备度：X级（置信度：高/中/低）
   评级依据：[D1-D6各子维度的关键判定依据]
   信息缺口影响：[...]

③ 技术互联度：X级（置信度：高/中/低）
   评级依据：[...]

④ 工艺可控度：X级（置信度：高/中/低）
   评级依据：[...]

⑤ 组织能力度：X级（置信度：高/中/低）
   评级依据：[...]

⑥ 管理变革度：X级（置信度：高/中/低）
   评级依据：[...]
```

**各维度评级标准（1-5级，按行业通用标准）：**
- 1级：几乎不具备；严重缺失，无法支撑AI项目
- 2级：初步具备；有方向但缺乏具体量化/系统化
- 3级：基本具备；满足AI项目启动的最低门槛
- 4级：良好具备；能支撑规模化，偶有小缺口
- 5级：完全具备；行业领先水平，可直接全量推进

### 【诊断结论】

```
准备度综合评分：X.X / 5
是否达到启动门槛：是/否（原因：...）

启动建议：
□ 全量启动：准备度充分，可按完整计划推进
□ 单台设备PoC：从1台设备小范围验证后再扩展
□ 先完成前置工作再启动：[具体前置工作清单，预计N个月]
```

### 【最关键瓶颈排序（5M1E根因分解）】

对最关键的2-3个瓶颈维度，进行5M1E根因分解：

```
瓶颈① [维度名称]（当前X级 → 需达Y级才能支撑目标）
对目标的影响：[不解决这个瓶颈，目标实现概率下降X%]
5M1E根因分解：
  人（Man）：[具体问题]
  机（Machine）：[具体问题]
  料（Material）：[具体问题]
  法（Method）：[具体问题]
  环（Environment）：[具体问题]

瓶颈② [维度名称]
...
```

### 【资料补充建议】

```
优先级1：补充[具体信息/文件]
  理由：将使[维度]评分置信度从低→高，可能将评级从X级提升至Y级
  预计收集难度：[低/中/高]

优先级2：补充[...]
优先级3：补充[...]
```

### 【目标-现实校准评估】⚠️ 条件触发

**触发条件（满足任一条即输出此模块）：**
- 用户雄心度分级为🟠激进或🔴变革性
- 用户OKR信心分数≤4
- 综合准备度评分<2.5 且目标改善幅度>15%

**未触发时：** 标注"✅ 目标与准备度匹配，无需校准"即可。

**触发时输出：**

```
【目标-现实校准评估】⚠️

目标雄心度：🟢保守/🟡适中/🟠激进/🔴变革性
当前综合准备度：X.X / 5级
支撑该目标所需最低准备度（基于WEF Lighthouse同类场景）：≥ X级
关键缺口维度：[差距最大的1-2个维度及缺口大小]

可达性判定：
□ ✅ 匹配：目标在当前准备度下，[X]个月内可合理期待实现
□ ⚠️ 偏激进：按当前准备度，预计仅能实现目标的[X%]，建议选择校准选项
□ ❌ 严重错位：当前准备度无法在设定时间内支撑该目标，必须修正

校准选项（⚠️或❌时必须提供三选一）：
  选项A · 调整目标值：匹配当前准备度的合理目标：___
  选项B · 扩展时间线：维持原目标，但时间节点延至___
  选项C · 分阶段设目标（推荐）：
    → 第一里程碑（90天）：___
    → 第二里程碑（6-12个月）：___
    → 最终目标（12-24个月）：___
```

## 信息不足时的降级处理

- 对该维度给出"低置信度"评分，标注"评级范围：X-Y级"
- 在补充建议中将该维度列为最高优先级
- 整体诊断结论标注置信度为"中"或"低"
- 不要因为信息不足而拒绝输出诊断——有什么分析什么
